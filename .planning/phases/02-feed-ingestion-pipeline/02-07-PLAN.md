    compute_confidence(feed_name: str, first_seen_dt: datetime, seen_in_feeds: int = 1) -> int:
    - quality_weight = QUALITY_WEIGHTS.get(feed_name, 10)
    - days_old = max(0, (datetime.now(timezone.utc) - first_seen_dt).days)
    - recency_bonus = max(0, 10 - days_old)  — linear decay, floor at 0 (Claude's Discretion from CONTEXT.md)
    - score = seen_in_feeds * 25 + recency_bonus + quality_weight
    - return min(100, score)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| main.py startup → all 5 feeds | main.py calls feed.run() synchronously on startup; any credential errors here are caught by BaseFeed.run() and written to Redis |
| APScheduler thread → pycti_client | Scheduled jobs share the same pycti_client instance across threads; pycti client must be thread-safe |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-07-01 | Denial of Service | All 5 feeds running simultaneously on startup | medium | mitigate | Feeds run sequentially (for loop) not concurrently in the immediate pass. After scheduler.start(), max_instances=1 per feed prevents concurrent scheduled runs of the same feed. |
| T-02-07-02 | Tampering | APScheduler concurrent run of same feed if start() called before immediate runs | medium | mitigate | Order in main(): immediate runs complete BEFORE scheduler.start(). max_instances=1 on each job as secondary protection per RESEARCH.md Pitfall 5. |
| T-02-07-03 | Information Disclosure | Confidence score formula reveals source quality weights | low | accept | QUALITY_WEIGHTS are metadata about feed reliability, not secrets. They appear in config.py; no mitigation needed. |
</threat_model>
